# Описание проекта `phase`

## Что делает проект

Проект валидирует таблицы `xlsx/csv` по набору правил из `YAML` и формирует отчет с ошибками (`errors.xlsx` или `errors.csv`).

Коротко: вы задаете правила в `rules.yml`, запускаете `validator.py`, на выходе получаете список нарушений по строкам.

---

## Как все работает (короткий сценарий)

1. Запуск через `validator.py` или `python -m src.xlsx_validator`.
2. В `cli.py` парсятся аргументы (`--input`, `--rules`, `--output`, `--sheet`, `--skip-rows`).
3. `io.py`:
   - читает входной файл (`DataReader`),
   - загружает YAML с правилами (`ConfigLoader`),
   - пишет итоговый отчет (`ReportWriter`).
4. `factory.py` превращает YAML-описания в объекты правил (`RuleFactory`).
5. `engine.py`:
   - проверяет, что нужные колонки есть в таблице,
   - запускает все правила по строкам/по всему DataFrame.
6. `rules.py` выполняет конкретные проверки (`value_in`, `required_if`, `date_between`, `time_between` и т.д.).
7. При нарушении создается объект ошибки (`ValidationErrorRow` из `models.py`).
8. `io.py` сохраняет все ошибки в выходной файл.

## Назначение каждого файла

### Корень проекта

- `validator.py`  
  Точка входа для запуска из корня. Просто вызывает `main()` из `src.xlsx_validator.cli`.

- `requirements.txt`  
  Зависимости проекта: `pandas`, `openpyxl`, `pyyaml`.

- `README.md`  
  Документация: установка, запуск, формат `rules.yml`, список поддерживаемых типов правил.

- `rules.yml`  
  Основной боевой набор правил валидации для реальных данных проекта.

- `rules_templates.yml`  
  Справочник шаблонов правил с комментариями: как быстро собрать новое правило и какие поля обязательны.

- `tft.yml`  
  Тестовый/укороченный набор правил для быстрого прогона.

- `DOLUT-2025_polnaya.xlsx`  
  Входной Excel-файл с исходными данными (пример/рабочий датасет).

- `RB-2405100-03-01_ProjectToOneFile_13-04-2026_07-05_03_UTC.xlsx`  
  Еще один входной Excel-файл с данными исследования.

- `errors.xlsx`  
  Сгенерированный отчет об ошибках после последнего запуска валидатора.

- `PROJECT_EXPLANATION.md`  
  Этот файл: обзор архитектуры и назначения файлов.

### Пакет `src`

- `src/__init__.py`  
  Маркер пакета `src` (чтобы корректно работал запуск как модуля).

### Пакет `src/xlsx_validator`

- `src/xlsx_validator/__init__.py`  
  Экспортирует `run_validation` как публичный API пакета.

- `src/xlsx_validator/__main__.py`  
  Точка входа для `python -m src.xlsx_validator`.

- `src/xlsx_validator/cli.py`  
  CLI-слой:
  - парсинг аргументов,
  - проверка существования файлов,
  - оркестрация чтения данных, создания правил, запуска валидации и записи отчета.

- `src/xlsx_validator/io.py`  
  Работа с файлами:
  - чтение YAML-конфига правил,
  - чтение `xlsx/csv` в DataFrame,
  - запись ошибок обратно в `xlsx/csv`.

- `src/xlsx_validator/factory.py`  
  Реестр типов правил (`_registry`) и фабрика, которая по `type` из YAML создает нужный класс правила.

- `src/xlsx_validator/engine.py`  
  Движок валидации:
  - проверяет схему (все ли нужные колонки есть),
  - запускает правила построчно и DataFrame-правила,
  - возвращает список ошибок.

- `src/xlsx_validator/rules.py`  
  Основная бизнес-логика правил:
  - базовый абстрактный класс `Rule`,
  - конкретные реализации (`ValueInRule`, `RequiredIfRule`, `DateBetweenRule`, `TimeBetweenRule`, `UniqueIfRule` и др.),
  - формирование `ValidationErrorRow` для нарушений.

- `src/xlsx_validator/conditions.py`  
  Выполняет условия `when` для условных правил (`equals`, `starts_with`, `less_than`, `number_between`, и т.д.).

- `src/xlsx_validator/models.py`  
  Модель строки ошибки (`ValidationErrorRow`) через `dataclass`.

- `src/xlsx_validator/utils.py`  
  Вспомогательные функции:
  - проверка пустых значений,
  - нормализация скаляров,
  - парсинг дат/времени,
  - сбор значений колонок в отчет.

### Служебные директории

- `.venv/`  
  Локальное виртуальное окружение Python с установленными пакетами.

- `__pycache__/`  
  Скомпилированные Python-байткоды (`.pyc`), создаются автоматически.

## Важные архитектурные идеи

- Правила описаны декларативно в YAML, а не хардкодятся в запускном скрипте.
- Добавление нового типа правила обычно требует:
  1) новый класс в `rules.py`,  
  2) регистрацию в `factory.py`,  
  3) использование в `rules.yml`.
- Условные проверки (`when`) вынесены отдельно в `conditions.py`, поэтому они переиспользуются всеми rule-классами.
- Отчет унифицирован: для любой ошибки сохраняются идентификаторы пациента, номер строки, правило, сообщение и значения полей.
